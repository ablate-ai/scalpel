# Scalpel

使用这个 skill 时，先做扫描，再做修改。不要在没有证据的情况下直接删除代码。

## 目标

把“代码瘦身”拆成两个阶段：

1. 用内置 Rust 扫描器二进制找出高概率重复/冗余区域。
2. 基于扫描结果，提出或执行低风险重构。

## 工作流

### 1. 确认扫描范围

优先确认或推断：

- 目标目录
- 是否只看某些语言或扩展名
- 是否排除 `vendor`、`dist`、`build`、`target`、`node_modules`、`.git`

如果用户没有给细节，默认扫描当前仓库，并排除上面的目录。

### 2. 运行扫描器

可执行扫描器位于 `skills/Scalpel/scripts/scalpel`。

直接运行：

```bash
./skills/Scalpel/scripts/scalpel \
  --path <target-path> \
  --format markdown
```

如果用户需要机器可读结果，使用：

```bash
./skills/Scalpel/scripts/scalpel \
  --path <target-path> \
  --format json
```

Rust 源码位于仓库级 `tools/scalpel-cli`，只有在需要改扫描器实现时才进入源码目录。安装 skill 时，不需要把源码一起带上。

### 3. 读取结果并分级

优先关注：

- `duplicate_files`：完全重复或归一化后重复的文件
- `duplicate_snippets`：跨文件或同文件的大段重复代码
- `boilerplate_hotspots`：同一文件内重复样板过多的区域

按下面顺序处理：

1. 完全重复文件
2. 跨文件重复片段
3. 同文件重复片段
4. 样板热点

### 4. 给出瘦身方案

方案要具体，优先给低风险动作：

- 抽公共函数
- 合并重复模块
- 用配置/表驱动替代重复分支
- 删除重复副本，统一引用单一实现
- 把重复常量或模板提升到共享位置

不要直接建议“删掉看起来没用的代码”，除非有明确引用证据。

### 5. 真正修改代码时

如果用户的目标是直接瘦身而不只是扫描：

- 先列出准备修改的重复点
- 优先处理收益高、影响面清晰的重复
- 修改后运行相关测试或最小验证
- 汇报哪些重复被合并、哪些还保留以及原因

## 输出要求

默认输出应包含：

1. 扫描范围
2. 最重要的重复/冗余发现
3. 建议的瘦身顺序
4. 如果已改代码，说明验证结果

## 结果解读原则

- 行数更多、文件更多、重复组更大的问题优先级更高。
- 生成代码、fixture、vendor 代码默认降权。
- 相同样板如果是框架约束，不要过度重构。
- 发现重复并不等于必须合并；只有在可读性和维护成本明确改善时才建议动刀。

## 示例

**示例 1**

用户请求：`帮我扫一下这个 Rust 项目里哪些模块重复实现了同样的校验逻辑，能瘦身的地方列出来`

动作：

- 运行扫描器扫描仓库
- 汇总重复片段
- 给出按收益排序的重构建议

**示例 2**

用户请求：`这个 services 目录太臃肿了，看看有没有 copy paste 代码，能的话顺手重构掉`

动作：

- 扫描 `services`
- 先锁定高重复片段
- 在证据充分时直接实施低风险合并
